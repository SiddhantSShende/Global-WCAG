# Accessibility Compliance Audit Platform — Technical Guide & Architecture

**Version:** 1.0
**Audience:** Engineering team building the platform
**Scope:** Automated + human-in-the-loop WCAG 2.0 / 2.1 / 2.2 (Levels A, AA, AAA) auditing for **Websites**, **Android apps**, and **iOS apps**, producing detailed `.docx` and `.xlsx` reports with screenshot evidence — modelled on the attached Jaypee Capital sample report.

---

## 0. Read this first — the accuracy contract

You asked for reports that are **100% accurate, legit, and contain no fabricated data.** That single requirement dictates the entire architecture, so it comes before everything else.

### The hard truth about automated accessibility testing

No tool on earth can automatically determine full WCAG conformance. This is not a tooling gap you can shop your way out of — it is inherent to the guidelines. Independent research (Deque, the ACT Rules community, TPGi) consistently puts **automatic detection at ~30–40% of WCAG issues.** The rest require human judgement or assistive-technology testing.

Our own generated dataset proves it. Of the **87** WCAG 2.2 success criteria:

| Testability | Count | Meaning |
|---|---|---|
| `auto` | **8** | A machine can decide it; a clean scan is real evidence toward a pass. |
| `semi` | **33** | A machine finds *some* violations, but a clean scan does **not** prove a pass. |
| `manual` | **46** | Not reliably machine-testable at all. Needs human / AT review. |

**Design consequence — the anti-fabrication rule:**

> The system may only emit **`Pass`** for a criterion when either (a) its `testability` is `auto` and every engine returned clean, **or** (b) a human reviewer explicitly signed off. In every other case it emits **`Needs Manual Review`**, never a silent `Pass`.

This is the opposite of what most "AI accessibility checkers" do (they mark everything green that they didn't flag red, which is fabrication). Your attached sample even contains this bug: criterion 1.1.1 is printed as `● Pass` while its own note says *"6 informative images missing alt text."* We build a **contradiction validator** (see §9.4) that makes that class of error impossible in our output.

### The iOS reality

You asked to "create emulators for Android and iOS in the backend." Android emulators run on Linux (with KVM). **iOS Simulators legally and technically only run on macOS** — Apple's SLA forbids running them elsewhere, and the binaries are macOS-only. There is no way around this. So the iOS scanner **must** run on Apple hardware (a Mac mini in the rack, or a macOS cloud runner such as MacStadium / AWS EC2 Mac / GitHub macOS runners). The architecture treats the iOS node as a separate, network-attached worker. Don't design as if a Linux container can emulate iPhones — it cannot, and pretending otherwise is exactly the "fabricated" territory you want to avoid.

### What "legit" therefore means for this platform

1. Every `Fail` carries machine-captured evidence: the offending selector/DOM node (or view hierarchy node), the computed values (e.g. contrast ratio, colors, sizes), the engine + rule ID that raised it, and a cropped, highlighted screenshot.
2. Every criterion we *cannot* prove is labelled honestly (`Needs Manual Review`) with the reason.
3. Nothing in a report is generated by a language model inventing facts. Text is templated from real measurements. (An LLM may *rephrase* remediation prose from a fixed knowledge base, but never invents a finding, a number, or a screenshot.)
4. Reproducibility: every scan stores its raw engine output (JSON) so any row in any report can be traced back to source.

## 1. What the platform does (product summary)

A user:

1. Chooses a **target type**: `Website`, `Android`, or `iOS`.
2. Provides the inputs for that type (see §3).
3. The platform runs the appropriate scanning pipeline (fully automated where possible; queued for human review where not).
4. The user downloads **6 reports** — one per WCAG combination — each in **both** `.docx` and `.xlsx`:

   | # | Report |
   |---|---|
   | 1 | WCAG 2.0 — Level A |
   | 2 | WCAG 2.0 — Level AAA |
   | 3 | WCAG 2.1 — Level A |
   | 4 | WCAG 2.1 — Level AAA |
   | 5 | WCAG 2.2 — Level A |
   | 6 | WCAG 2.2 — Level AAA |

   > A "Level AAA" report is cumulative — it contains A + AA + AAA criteria, matching how conformance levels nest. If you also want the intermediate AA-only cut, the engine already computes it (§5.6); it's just a matter of exposing 9 combinations instead of 6. Keep the config-driven `REPORT_MATRIX` so this is a one-line change.

Each report mirrors the attached sample's structure (§8): document control, confidentiality statement, scope, methodology, executive summary, full compliance checklist, detailed per-issue observations **with embedded highlighted screenshots**, risk evaluation, and prioritized recommendations.

**Key principles**

- **Workers are stateless and pluggable.** Each scan engine is wrapped behind a common interface (`run(target) -> list[RawIssue]`). Adding a new engine never touches the rest of the system.
- **The canonical Findings schema (§4) is the contract.** Every engine, for every platform, maps *onto* it. The report builder only ever sees canonical findings — it doesn't know or care whether they came from axe or Appium.
- **The WCAG dataset (§ wcag_data) is the source of truth** for which criteria exist per version/level and how testable each is.
- **Secrets never touch disk in plaintext.** App credentials and authenticated-scan logins go through a vault (§10).

## 3. User flow & inputs per target type

### 3.1 Website
```
User submits:  https://example.com   (+ optional auth: login URL, username, password/secret)
        │
        ▼
[1] Subdomain discovery   subfinder + amass + crt.sh  →  list of live hosts
        │
        ▼
[2] Crawl each host        katana / Playwright crawler →  page inventory (deduped, capped)
        │
        ▼
[3] Per-page scan          axe-core + Pa11y + Lighthouse + IBM Equal Access (parallel)
        │
        ▼
[4] Evidence capture       Playwright element screenshots w/ highlight box
        │
        ▼
[5] Normalize + dedupe     → canonical Findings, grouped by criterion & rule
        │
        ▼
[6] Human-review queue     all `semi`/`manual` criteria surfaced for sign-off (optional gate)
        │
        ▼
[7] Report build           6 combos × (docx + xlsx)
```

**Inputs:** domain URL (required); scope controls (include/exclude subdomains, max pages per host, crawl depth); optional authenticated-session credentials; optional rate-limit / robots.txt honoring toggle. **Always require the user to confirm they own or are authorized to test the domain** (§10.4 — legal gate).

### 3.2 Android
**Inputs:** the `.apk`/`.aab` file upload (or a Play Store URL for reference builds you're authorized to test); optional test-account credentials; optional deep-link list / login steps.

Flow: spin an Android Virtual Device (AVD) on a KVM-enabled Linux worker → install APK → drive the app with Appium/UIAutomator → at each meaningful screen run Google's **Accessibility Test Framework for Android (ATF)** against the live view hierarchy → capture screenshot + node bounds → normalize.

### 3.3 iOS
**Inputs:** the `.app`/`.ipa` (simulator build for automated audits; a real-device build needs a physical device farm) + optional test-account credentials.

Flow (on the **macOS** worker): boot an iOS Simulator via `xcrun simctl` → install the app → drive with Appium/XCUITest → at each screen call XCUITest's **`performAccessibilityAudit()`** (Xcode 15+/iOS 17+) which returns audit issues (contrast, hit-region size, element description, clipped text, trait conflicts) → capture screenshot + element frame → normalize.

> Automated mobile a11y coverage is even thinner than web. ATF and XCUITest audits cover a real but limited set (labels, contrast, target size, some traits). Everything else is `Needs Manual Review`. Report this honestly.

## 4. Canonical Findings schema (the contract)

Everything normalizes to this. Store as JSONB in Postgres and as raw JSON in object storage. Defined in code at `backend/app/models.py`.

```jsonc
Finding {
  "finding_id": "uuid",
  "job_id": "uuid",
  "target_type": "web | android | ios",
  "target_ref": "https://... | screen-id | activity-name",

  // WCAG mapping — from the wcag_criteria.json registry
  "sc_num": "1.4.3",
  "sc_name": "Contrast (Minimum)",
  "level": "AA",
  "principle": "Perceivable",
  "wcag_versions": ["2.0", "2.1", "2.2"],

  // Verdict — the anti-fabrication core
  "status": "fail | pass | needs_manual_review | not_applicable | partial",
  "confidence": "high | medium | low",
  "auto_decidable": true,           // == criterion.testability == 'auto'

  // Provenance — every finding traces to a real detection
  "engine": "axe-core | pa11y | lighthouse | ibm-equal-access | android-atf | ios-xcuitest",
  "engine_rule_id": "color-contrast",
  "impact": "blocker | serious | moderate | minor",

  // Evidence
  "selector": "main > div.card > a",     // CSS selector or view-hierarchy path
  "html_snippet": "<a href=...>...</a>", // or native element descriptor
  "computed": {                          // real measured values, engine-provided
    "contrast_ratio": 2.32,
    "fg": "#a5a5a5", "bg": "#ffffff",
    "font_size_px": 14, "font_weight": 400,
    "required_ratio": 4.5
  },
  "screenshot_key": "s3://.../job/finding.png",   // cropped + highlighted
  "page_screenshot_key": "s3://.../job/page.png",  // full page/screen context
  "occurrences": 8,
  "locations": [{"url": "...", "count": 2}, ...],

  "description": "Text from the engine's rule metadata (not invented).",
  "remediation": "Templated from a fixed remediation KB keyed by rule/criterion.",
  "raw_engine_payload_key": "s3://.../job/raw/axe.json"   // full traceability
}
```

**Deduplication key:** `(sc_num, engine_rule_id, normalized_selector)` across pages → merges into one finding with `occurrences` and `locations` rolled up, exactly like the sample's "Issue" cards ("Total Occurrences: 69, across 6 pages").

**Status precedence** when multiple engines disagree on the same criterion: `fail` > `partial` > `needs_manual_review` > `pass` > `not_applicable`. A single confirmed `fail` from any credible engine wins. Never let one engine's silence upgrade another's `fail`.

## 5. Web scanning pipeline (detailed)

### 5.1 Subdomain discovery (`scanners/web/subdomains.py`)
OSINT tools, run in parallel, results unioned and liveness-checked:

- **subfinder** (ProjectDiscovery) — passive, fast, many sources.
- **amass** (OWASP) — passive + optional active enumeration.
- **crt.sh** query — certificate-transparency logs (`https://crt.sh/?q=%25.example.com&output=json`).
- **assetfinder** (optional) — extra sources.

Liveness: probe with **httpx** (ProjectDiscovery) to keep only responsive hosts and capture status/title/tech. Respect the authorization gate — only enumerate domains the user attested they own.

### 5.2 Crawling (`scanners/web/crawler.py`)
- **katana** (ProjectDiscovery) for fast, headless-aware crawling **or** a Playwright-driven crawler when you need JS-rendered SPA routes and authenticated sessions.
- Config: `max_pages_per_host`, `max_depth`, same-origin restriction, URL de-dup by normalized path, exclude patterns (logout, mailto, tel, file downloads), sitemap.xml seeding.
- Output: a page inventory. **Cap it** (e.g. 100 pages/host default) — accessibility issues are template-driven, so scanning every one of 10,000 near-identical product pages wastes compute; sample representative templates and record the sampling in the report's methodology.

### 5.3 Engine orchestration (`scanners/web/engines.py`)
Each engine implements:
```python
class Engine(Protocol):
    name: str
    def scan(self, page: PageContext) -> list[RawIssue]: ...
```

Engines (run in parallel per page):

| Engine | How to run | WCAG coverage | Notes |
|---|---|---|---|
| **axe-core** | `@axe-core/playwright` injected into the live Playwright page | 2.0/2.1/2.2 A & AA + best-practice, tagged `wcag2a`, `wcag2aa`, `wcag21a`, `wcag22aa`, etc. | Industry standard; low false positives. Primary engine. |
| **Pa11y** | Node CLI/`pa11y` module; runner `htmlcs` | HTML_CodeSniffer standards `WCAG2A`, `WCAG2AA`, **`WCAG2AAA`** | The main source of *some* AAA automated checks. |
| **Lighthouse** | `lighthouse` Node module, `--only-categories=accessibility` | axe-based subset + audits | Gives a score + a few extra audits. |
| **IBM Equal Access** | `accessibility-checker` Node module | ACT-rules aligned, 2.0/2.1/2.2 | Strong ruleset, good for cross-checking. |

Run all four; keep each engine's raw JSON in object storage; the normalizer reconciles them. Using multiple engines both raises recall and lets you flag disagreements for review.

### 5.4 Evidence capture (`scanners/web/evidence.py`)
For every `fail`, reproduce the sample's highlighted screenshots with Playwright:
```python
el = page.locator(selector)
box = el.bounding_box()
# 1. full-page screenshot for context
page.screenshot(path=page_png, full_page=True)
# 2. draw a highlight rectangle over the element, then crop with padding
page.evaluate("""(sel) => {
    const n = document.querySelector(sel);
    const o = document.createElement('div');
    Object.assign(o.style, {position:'absolute', outline:'3px solid #e11', 
        zIndex:2147483647, pointerEvents:'none'});
    const r = n.getBoundingClientRect();
    Object.assign(o.style, {left:(r.left+scrollX)+'px', top:(r.top+scrollY)+'px',
        width:r.width+'px', height:r.height+'px'});
    document.body.appendChild(o);
}""", selector)
el.scroll_into_view_if_needed()
page.screenshot(path=finding_png, clip=padded(box))
```
Store both keys on the finding. The report embeds the cropped, highlighted shot under "Visual Evidence" and can link the full-page context.

### 5.5 Normalization (`scanners/web/normalize.py`)
- Map each engine's rule ID → WCAG `sc_num` via a maintained **rule→criterion map** (axe already tags criteria; HTML_CodeSniffer codes embed the SC; IBM maps to ACT/WCAG). Keep this map in `wcag_data/rule_maps/`.
- Merge duplicates by the dedup key (§4).
- Assign `impact` from the engine (axe's impact field) or a severity table keyed by criterion.
- Assign `status`/`confidence` per the anti-fabrication rule (§0, §9).

### 5.6 Scoping to the 6 report combinations (`reporting/common.py`)
Given all findings + the WCAG registry, for each `(version, level)`:
1. Select in-scope criteria: `version in c.versions AND c.level in level_scope(level)` (helper already in `build_wcag_json.py`).
2. For each criterion compute its row status:
   - any in-scope `fail` finding → **Fail**
   - else if `testability == auto` and engines ran clean → **Pass**
   - else if reviewer signed off → their verdict
   - else → **Needs Manual Review**
3. Emit the checklist + detailed issue cards for that combo.

`REPORT_MATRIX = [("2.0","A"),("2.0","AAA"),("2.1","A"),("2.1","AAA"),("2.2","A"),("2.2","AAA")]` — config-driven so adding AA cuts is trivial.

## 6. Android scanning pipeline

- **Emulator:** Android SDK `emulator` + a headless AVD on a KVM Linux host (`-no-window -gpu swiftshader_indirect`). One AVD per job for isolation; snapshot a clean image to reset fast.
- **Driver:** **Appium** (UiAutomator2 driver) installs the APK, logs in with provided credentials, and walks screens (either a provided click-path or a crawler that taps focusable elements breadth-first with loop detection).
- **Audit engine:** Google **Accessibility Test Framework for Android (ATF)** — the same engine behind Play Console's pre-launch report and Accessibility Scanner. Run its checks against each screen's view hierarchy. It flags: unlabeled/`contentDescription`-missing controls, low text/image contrast, touch targets < 48dp, duplicate descriptions, clickable-span issues, etc.
- **Evidence:** `driver.get_screenshot_as_file()` + the node's `bounds` rect → crop + highlight (same technique as web).
- **Mapping:** ATF check classes → WCAG criteria (e.g. `TouchTargetSizeCheck` → 2.5.8/2.5.5; `TextContrastCheck` → 1.4.3/1.4.6; `SpeakableTextPresentCheck` → 1.1.1/4.1.2). Keep this map in `wcag_data/rule_maps/android_atf.json`.
- Everything ATF can't judge → `Needs Manual Review` (TalkBack manual pass recommended in the report).

## 7. iOS scanning pipeline (macOS worker only)

- **Simulator:** boot via `xcrun simctl boot <UDID>`; install with `simctl install`; launch with `simctl launch`.
- **Driver:** **Appium** (XCUITest driver) or a native XCUITest UI-test target for deeper access.
- **Audit engine:** XCUITest **`performAccessibilityAudit()`** (`XCUIApplication().performAccessibilityAudit()`), auditing for contrast, element description, hit-region size, clipped text, trait conflicts, dynamic-type support. Returns structured issues you catch/aggregate.
- **Evidence:** `XCUIScreen.main.screenshot()` + element `frame` → crop + highlight.
- **Mapping:** audit issue types → WCAG criteria; store in `wcag_data/rule_maps/ios_xcuitest.json`.
- Physical-device coverage (real VoiceOver rotor behavior, gestures) is manual — note it.

> **Provisioning note:** iOS audits need a simulator build of the app. If the client can only provide an App Store `.ipa`, automated simulator auditing may not be possible without a resigned/dev build — capture this as an intake requirement.

## 8. Report generation (matches the attached sample)

Two generators, both driven by the same in-scope findings for a combo:

- **`reporting/docx_report.py`** (python-docx) → the narrative report.
- **`reporting/xlsx_report.py`** (openpyxl) → the machine-friendly workbook (checklist sheet + findings sheet + summary sheet, with conditional formatting and embedded thumbnails).

### 8.1 DOCX section map (from the sample)
1. **Cover** — standard, level & version, target, date, auditor org.
2. **Document Control** — table (report name, version, auditor, reviewer, client, vendor, standard, period, environment).
3. **Statement of Confidentiality and Limitations** — templated; **include the automation-limits disclaimer** (§0) here.
4. **Timelines** — table.
5. **Scope** — table (project, activity, URL/app, standard).
6. **Risk Rating** — Blocker/Serious/Moderate/Minor definitions.
7. **Testing Methodology** — phases, categories, tools used (list the actual engines that ran).
8. **Information Gathering** — target description, tech stack (from httpx), pages/screens covered.
9. **Executive Summary** — counts of Fail / Needs-Review; **Coverage Summary** table (criteria evaluated vs total for this combo — pull straight from the registry); confirmed-failures severity tally; conformance-status table (Fail/Partial rows with issue refs).
10. **Full WCAG Compliance Checklist** — *every* in-scope criterion with status glyph (● Fail / ▲ Partial / ✔ Pass / ⚠ Needs Manual Review / — N/A), issue ref, and note. Grouped by principle → guideline like the sample.
11. **Detailed Observations** — one card per confirmed issue: criterion, level, impact, test URL(s), total occurrences, element locations, description, user impact, **Visual Evidence (embedded highlighted screenshot)**, remediation steps.
12. **Risk Evaluation and Conclusion** — severity table + overall exposure.
13. **Recommendations** — prioritized (P1 blocker → P3), derived from the failing criteria.
14. **Prepared By** footer.

> The single change from the sample: wherever the sample silently prints `Pass` on an unverifiable criterion, we print `⚠ Needs Manual Review`. That is the legitimacy upgrade.

### 8.2 python-docx implementation notes
- Build a reusable `DocxTheme` (fonts, brand color, table styles) so all 6 reports are consistent.
- Status glyphs via colored runs; severity via cell shading.
- Embed screenshots with `document.add_picture(png_path, width=Inches(6))`; keep images ≤ ~1600px wide to control file size (the sample was 15 MB — downscale evidence shots).
- Tables: set explicit column widths; repeat header rows for long checklists.
- Generate a real Table of Contents field, or a static one if you don't want users to "update fields."

### 8.3 xlsx workbook
- **Summary** sheet: combo, coverage %, severity counts, overall exposure.
- **Checklist** sheet: one row per in-scope criterion (num, name, level, principle, status, issue ref, note) with conditional formatting by status.
- **Findings** sheet: one row per finding (criterion, impact, engine, rule, selector, computed values, occurrences, locations, screenshot thumbnail via `openpyxl.drawing.image.Image`).
- Freeze panes, autofilter, and a legend.

## 9. Anti-fabrication & accuracy engineering

### 9.1 Status derivation (single source, no shortcuts)
Implement one pure function `derive_status(criterion, findings, review)` used everywhere. It encodes §0's rule. No other code may set a status.

### 9.2 Confidence tiers
- `high`: `auto` criterion, engine-measured (e.g. contrast 2.32:1). 
- `medium`: `semi` criterion with a concrete violation.
- `low`: heuristic detections — always route to human review before they can become a `Pass`.

### 9.3 Multi-engine reconciliation
Cross-check engines. Agreement → higher confidence. Disagreement → flag for review, never auto-resolve to the rosier verdict.

### 9.4 Contradiction validator (mandatory pre-report gate)
Before any report is written, run assertions over the combo's rows:
- No row may be `Pass` while a `fail` finding exists for its criterion.
- No `auto`-only Pass may exist for a criterion whose `testability != auto`.
- Every `Fail` must have ≥1 finding with evidence (`screenshot_key` and `computed` or `html_snippet`).
- Every `occurrences` count must equal the sum of `locations[].count`.
If any assertion fails, the build aborts with a diagnostic — a contradictory report never ships.

### 9.5 No LLM-invented facts
LLMs may only (a) rephrase remediation text from a fixed KB, and (b) draft the executive-summary prose from the computed numbers passed in as structured facts. They never see a blank page and never produce findings, counts, or evidence. Keep temperature low and pass numbers as literals the model must echo.

## 10. Security, secrets & legal

1. **Authorization gate.** Before any web scan, require an explicit attestation that the user owns/represents the target. Log it. Subdomain enumeration + crawling of a domain you don't control can be unlawful.
2. **Secret vault.** App credentials and login secrets go to HashiCorp Vault (or cloud KMS-backed secrets). They're injected into the worker at runtime, never written to disk, never logged, and purged after the job.
3. **Isolation.** Each scan job runs in an ephemeral container/AVD/simulator, torn down after. Malicious target sites shouldn't be able to pivot.
4. **PII in screenshots.** Authenticated scans may capture personal data. Offer redaction and short retention; make report storage access-controlled.
5. **Rate limiting / robots.** Provide a "polite mode" and honor client-configured limits so scans don't take down the target.
6. **Audit trail.** Every job records who ran it, against what, when, with which engine versions.

## 11. Deployment & infrastructure

- **Containers:** API, web-scan worker, android-scan worker as Docker images; iOS worker as a provisioned macOS host (not containerizable) joined to the same queue.
- **Queue:** Redis + RQ or Celery. Long scans are async; the frontend polls/subscribes for status.
- **Storage:** PostgreSQL (metadata/findings) + S3/MinIO (raw JSON, screenshots, reports).
- **Android worker** needs `/dev/kvm` (bare-metal or nested-virt VM). GPU optional (`swiftshader` software rendering works headless).
- **iOS worker:** Mac mini rack or EC2 Mac / MacStadium; keep Xcode + simulators pinned to known versions for reproducibility.
- **Scaling:** workers are horizontal; the AVD/simulator boot time dominates mobile jobs, so keep warm pools.
- **Versioning:** pin axe-core, Pa11y/HTML_CodeSniffer, Lighthouse, IBM checker, ATF, and Xcode versions per job and record them in the report — results must be reproducible.

## 14. Build roadmap (phased)

**Phase 1 — Web MVP (highest value, fully automatable path).**
Registry (done) → subdomain+crawl → axe-only scan → normalize → docx+xlsx for the 6 combos → contradiction validator. Ship this first; it already produces legit reports.

**Phase 2 — Multi-engine + evidence.**
Add Pa11y (AAA), Lighthouse, IBM; add highlighted-screenshot evidence; reconciliation.

**Phase 3 — Human-review workbench.**
UI to work the `semi`/`manual` queue; reviewer sign-off feeds `derive_status`. This is what turns "partial automated scan" into "defensible audit."

**Phase 4 — Android.**
KVM worker, AVD pool, Appium + ATF, mapping, evidence.

**Phase 5 — iOS (macOS node).**
Simulator pool, XCUITest audits, mapping, evidence.

**Phase 6 — Scale & polish.**
Warm pools, auth-session scanning, PII redaction, report theming, PDF export.

## 15. Known limitations to state plainly (put these in every report)

- Automated tooling detects a minority of WCAG issues; a clean automated scan is **not** a conformance claim.
- `Needs Manual Review` rows require a qualified human + assistive-technology pass to close.
- AAA is largely manual; AAA reports will be review-heavy by design.
- Mobile automated audits (ATF, XCUITest) cover a limited rule set.
- Results reflect the target's state at scan time and the pages/screens actually reached.
- iOS auditing requires a simulator-installable build and Apple hardware.

Honesty here is not a weakness — it is what makes the report legally and professionally defensible, and it's exactly what "no fabricated data" requires.
